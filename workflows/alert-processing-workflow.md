# Alert Processing Workflow

## Overview

This workflow covers how security alerts generated by Wazuh are received by Shuffle SOAR, parsed, and enriched with threat intelligence before being passed downstream for case creation or notification.

## Flow Diagram

```
Wazuh Manager
     |
     | (Webhook - HTTP POST)
     v
Shuffle SOAR Trigger
     |
     v
Alert Parsing (Extract fields)
     |
     v
IP Enrichment
  +---> VirusTotal   (malicious score, detection ratio)
  +---> AbuseIPDB    (abuse confidence %, ISP, country)
  +---> GreyNoise    (noise classification, RIOT flag)
     |
     v
Severity Classification
     |
     v
Route to Incident Management + Teams Notification
```

## Trigger

Wazuh is configured to forward alerts to Shuffle via a webhook integration. The trigger fires whenever a Wazuh alert meets or exceeds the configured rule level threshold (level >= 7 by default).

The webhook payload is a JSON object delivered via HTTP POST to the Shuffle workflow webhook endpoint.

## Alert Fields Extracted

| Field | Source | Description |
|---|---|---|
| `rule.id` | Wazuh | Unique rule identifier |
| `rule.description` | Wazuh | Human-readable alert description |
| `rule.level` | Wazuh | Severity level (1 to 15) |
| `rule.groups` | Wazuh | Rule category tags (e.g. authentication_failed) |
| `agent.name` | Wazuh | Hostname of the source agent |
| `agent.ip` | Wazuh | IP address of the source agent |
| `data.srcip` | Wazuh | Source IP involved in the event |
| `data.dstip` | Wazuh | Destination IP involved in the event |
| `timestamp` | Wazuh | UTC timestamp of the alert |
| `location` | Wazuh | Log source path or integration name |

## IP Enrichment Steps

### VirusTotal

The source IP is submitted to the VirusTotal API. The following fields are extracted from the response:

| Field | Description |
|---|---|
| `malicious` | Number of engines flagging the IP as malicious |
| `suspicious` | Number of engines flagging the IP as suspicious |
| `harmless` | Number of engines reporting the IP as clean |
| `reputation` | VirusTotal community reputation score |

### AbuseIPDB

The source IP is submitted to the AbuseIPDB API. The following fields are extracted:

| Field | Description |
|---|---|
| `abuseConfidenceScore` | Percentage confidence the IP is abusive (0 to 100) |
| `countryCode` | Country of origin |
| `isp` | Internet service provider |
| `totalReports` | Total number of abuse reports on record |
| `lastReportedAt` | Timestamp of most recent abuse report |

### GreyNoise

The source IP is submitted to the GreyNoise API. The following fields are extracted:

| Field | Description |
|---|---|
| `noise` | Whether the IP is a known internet scanner |
| `riot` | Whether the IP belongs to a benign service (e.g. Google, Cloudflare) |
| `classification` | malicious / benign / unknown |
| `name` | Actor or organisation name if known |

## Output

Once enrichment is complete, the workflow produces an enriched alert object containing the original Wazuh fields plus all threat intelligence data. This object is passed to:

- The incident management workflow to create a DFIR IRIS case
- The Teams notification workflow to alert on-call analysts

## Error Handling

| Scenario | Behaviour |
|---|---|
| Enrichment API timeout | Workflow continues with partial enrichment; missing fields noted in case |
| No source IP in alert | IP enrichment steps are skipped |
| Shuffle webhook unreachable | Wazuh retries delivery up to 3 times |
| API key invalid or rate limited | Alert is flagged and forwarded without enrichment data |
